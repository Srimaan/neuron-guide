Multi-Agent
Architecture

One agent can't do everything well. A Supervisor agent routes tasks to specialised sub-agents — one for GitHub, one for Jira, one for Slack, one for Browser automation, one for Knowledge Base queries. Parallel execution, specialised prompts, better results.

⏱ 4–5 hours 🧠 5 sub-agents ⚡ Parallel execution 💰 ~$150/mo
🎯
Goal: Replace the single general-purpose agent with a Supervisor + 5 Sub-Agent architecture. The Supervisor classifies intent and routes to the right specialist. Sub-agents run in parallel for multi-part requests.
🧱
Deliverable: A SupervisorAgent that receives user requests, calls up to 3 sub-agents in parallel using asyncio.gather(), and synthesises a single coherent response. A prompt that previously took 12 seconds now takes 5 — parallel execution nearly halves latency for complex requests.

Why Multi-Agent?

A single general agent is like one person doing every job. It works for simple tasks but becomes a bottleneck for complex ones. Multi-agent gives each specialist the right context, the right tools, and the right instructions — and runs them simultaneously.

Figure 18.1 — Single Agent vs Multi-Agent: Complexity Handling

The Supervisor + 5 Sub-Agent Architecture

The Supervisor Agent receives every request, classifies intent using a lightweight model (Haiku), and routes sub-tasks to the right specialists. Results come back, and the Supervisor synthesises the final response.

Figure 18.2 — Supervisor + 5 Sub-Agent Full Architecture

Intent Classification — The Routing Brain

The Supervisor uses Claude Haiku (fast + cheap) to classify the intent of each request and select which sub-agents to call. Classification adds only ~200ms and saves the cost of calling full Sonnet for routing decisions.

Figure 18.3 — Intent Classification: Haiku Routes, Sonnet Executes

The multi-agent system adds a new layer to neuron-mcp-client. The existing single-agent code stays intact — the new SupervisorAgent wraps it and adds routing logic.

Step 18.3 — Create the SupervisorAgent
Orchestrates routing, parallel execution, and response synthesis
✨
Claude Prompt:

"Create mcp_client/services/supervisor.py with class SupervisorAgent.

__init__: initialise all 5 sub-agents, create a Haiku client for classification, create a Sonnet client for synthesis.

async _classify_intent(prompt: str) → list[str]: call Haiku with system='Classify this request. Return JSON list of agents needed. Options: github, jira, slack, browser, knowledge_base. Max 3. Examples: find PRs → [\"github\"], sprint + notify → [\"jira\",\"slack\"]'. Parse JSON response. On error return [\"github\"] as fallback.

async _run_agents_parallel(tasks: dict[str, str]) → list[SubAgentResult]: use asyncio.gather(*[agent.run(task) for name,task in tasks.items() if name in self._agents]). Returns results list.

async invoke_streaming(prompt: str, session_id: str, history: list) → AsyncGenerator[tuple, None]: (1) yield ('thinking', 'Classifying request...'), (2) classify intent, (3) yield ('thinking', f'Routing to: {agents}'), (4) decompose prompt into per-agent sub-tasks using Haiku, (5) run agents in parallel, (6) yield tool_use events for each agent that ran, (7) synthesise with Sonnet using all results as context, (8) yield text chunks, (9) yield complete event."

✅ All of these must be true:

  1. Single-agent mode still works — multi_agent: false behaves identically to before
  2. A GitHub-only prompt shows only GitHub tools in SSE events
  3. A cross-domain prompt (Jira + Slack) shows both agent names in the thinking event
  4. Cross-domain elapsed time ≤ slowest single agent (proves parallel execution)
  5. The Streamlit UI has a "Multi-Agent Mode" toggle in the sidebar
  6. Each sub-agent has a distinct system prompt visible in its class definition

What You Have Learned

Specialisation beats generalism
A GitHub agent with GitHub-only tools and a GitHub-expert prompt outperforms a general agent given the same request. Focused context = better reasoning = better results.
Haiku routes, Sonnet executes
Use cheap/fast models for classification. Use capable models for actual reasoning. The 200ms Haiku routing call saves 2 seconds of Sonnet thinking about which tools to pick.
asyncio.gather() is free parallelism
Running 3 agents in parallel with asyncio.gather() costs the same as running 1 — you're waiting on network IO anyway. Total time = max(agent times), not sum.
Backwards compatibility matters
The single-agent mode stays intact. Multi-agent is an opt-in flag. Users who depend on the existing behaviour aren't broken. Always extend, never replace working systems.
