The Dispatcher
Pattern

Refactor from 30 individual tools to 6 dispatcher tools. The AI discovers available operations at runtime instead of pre-loading everything. Result: ~70% fewer tokens on every single request — making Neuron dramatically faster and cheaper.

⏱ 3–4 hours 💰 Saves ~70% tokens 🏗 Architecture refactor 🔧 No new features
🎯
Goal: Refactor neuron-mcp-server from 30 individual MCP tools to 6 dispatcher tools — 2 per integration (github, jira, slack). Same capabilities. Dramatically lower token usage on every request.
🧱
Deliverable: Health check shows tools_registered: 8 (down from 32). All existing functionality still works. The AI calls list_github_ops() first to discover available operations, then github_tool(op, args) to execute.

The Problem: Token Explosion

Every message you send to Neuron includes the schema of every registered tool. With 30 tools, that is thousands of tokens wasted before Claude even reads your message. As you add more tools, the cost grows linearly.

Figure 11.1 — Token Cost Per Request: Before vs After Dispatcher

How the Dispatcher Pattern Works

Instead of one MCP tool per operation, we have two tools per category: a discovery tool and an execution tool. The AI calls discovery first to see the menu, then calls execution with a specific operation name.

Refactoring Approach — What Changes

The refactor is surgical. The underlying tool functions keep all their logic — we just rename them with an underscore prefix and stop registering them as MCP tools. A new dispatcher function routes between them.

Refactor to:
1. Rename all existing async functions — remove @mcp.tool() decorator, add underscore prefix: github_search_prs → _github_search_prs. Keep ALL logic exactly the same.
2. Add @mcp.tool() async def list_github_ops() → dict — docstring: 'Returns all available GitHub operations. Call this before using github_tool() to see what operations are available and their parameters.' Returns static dict: {operations: [{name, description, required_params:[], optional_params:[]}]} for all 12 operations. No API calls — just a static catalog.
3. Add @mcp.tool() async def github_tool(op: str, args: dict = {}) → dict — docstring: 'Execute a GitHub operation. Call list_github_ops() first to see available operations.' Build handlers dict mapping op strings to underscore-prefixed functions. If op not found return {error:true, message: f'Unknown op: {op}. Call list_github_ops() to see available operations.'}. Call handler(**args) in try/except with format_error().
4. Update register_github_tools(mcp) to register ONLY list_github_ops and github_tool.

Show complete refactored github_tools.py."

What You Have Learned

Token cost scales with tool count
Every tool schema is sent with every message. 30 tools × 150 tokens = 4,500 tokens of overhead per request. The dispatcher pattern breaks this linear scaling.
Discovery-then-execute is smarter
The AI calling list_ops() then tool(op=...) is more intelligent than pre-loading everything. It mirrors how a human checks a menu before ordering.
Adding operations is now free
To add a new GitHub operation: add one entry to the catalog dict and one line to the handlers map. Zero new MCP tools. Zero additional tokens per message.
Underscore prefix = internal function
The underscore prefix is a Python convention meaning "not for external use". The AI cannot call _github_search_prs directly — only through the dispatcher.
