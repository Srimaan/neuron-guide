Knowledge Base
& RAG

Give the AI access to your company's internal knowledge — runbooks, architecture docs, API specs, onboarding guides. Upload documents to S3, let Bedrock index them automatically, and ask the AI questions that get grounded, cited answers from real documents.

⏱ 3–4 hours 📚 Bedrock Knowledge Base 🔍 Semantic search 📎 Cited answers
🎯
Goal: Create an Amazon Bedrock Knowledge Base backed by the S3 bucket from Chapter 2. Upload internal docs, trigger an ingestion sync, and wire up two new MCP tools — kb_search and kb_retrieve — that return chunked, cited passages the AI can reference.
🧱
Deliverable: Asking "How do we handle API rate limits?" returns an answer citing the exact document, page, and passage from your internal runbook. The Knowledge Base Agent in Chapter 18's multi-agent system is now fully functional.

What RAG Is and Why It Matters

RAG (Retrieval-Augmented Generation) is the pattern of fetching relevant document chunks at query time and including them in the AI's context. This grounds the AI's answer in real documents instead of its training data — eliminating hallucination on company-specific knowledge.

Figure 19.1 — RAG vs Pure LLM: Why Documents Beat Training Data

Bedrock Knowledge Base Architecture

Amazon Bedrock Knowledge Base is a fully managed RAG pipeline. You upload documents to S3, Bedrock chunks them, embeds each chunk with Titan Embeddings V2, and stores the vectors in a managed OpenSearch cluster. At query time, Bedrock does the retrieval and returns ranked passages.

Figure 19.2 — Bedrock Knowledge Base: Ingestion + Query Pipelines

What Documents to Upload

The Knowledge Base is most useful when it contains documents your team references repeatedly but that aren't in GitHub — runbooks, architecture decisions, API specs, onboarding guides, post-mortems.

The Two Knowledge Base MCP Tools

We add two tools to the MCP server. kb_search is the fast semantic search for discovery. kb_retrieve is the detailed retrieval for reading full passages with citations.

Build Steps

Step 19.1 — Create the Bedrock Knowledge Base
One-time setup in the AWS Console — 10 minutes
  1. Go to AWS Console → Amazon Bedrock → Knowledge Bases → Create
  2. Name: neuron-knowledge-base
  3. IAM role: create a new role (Bedrock will create it automatically)
  4. Data source: Amazon S3 → select your neuron-platform-docs-{account_id} bucket
  5. Embeddings model: Titan Text Embeddings V2
  6. Vector database: Amazon OpenSearch Serverless (managed, no setup)
  7. Click Create Knowledge Base — takes ~3 minutes
  8. Note the Knowledge Base ID — you'll need it in the .env file
Step 19.2 — Upload and Sync Documents
Upload your internal docs to S3 and trigger the first ingestion
bashUpload docs and sync
# Upload your internal documents to S3
aws s3 cp docs/ s3://neuron-platform-docs-$ACCOUNT_ID/knowledge-base/ \
  --recursive \
  --profile neuron-dev

# Start the ingestion job
aws bedrock-agent start-ingestion-job \
  --knowledge-base-id YOUR_KB_ID \
  --data-source-id YOUR_DATA_SOURCE_ID \
  --region us-east-1 \
  --profile neuron-dev

# Check ingestion status (~2-5 min for 10-20 docs)
aws bedrock-agent get-ingestion-job \
  --knowledge-base-id YOUR_KB_ID \
  --data-source-id YOUR_DATA_SOURCE_ID \
  --ingestion-job-id YOUR_JOB_ID \
  --query "ingestionJob.status"
# Expected: "COMPLETE"
Step 19.3 — Add KB Tools to MCP Server
New kb_tools.py with kb_search and kb_retrieve

"Create mcp_server/tools/kb_tools.py for neuron-mcp-server. Uses boto3 bedrock-agent-runtime client. Settings: KB_ID from settings.knowledge_base_id, AWS region from settings.

1. @mcp.tool() async def kb_search(query: str, max_results: int = 5) → dict: calls bedrock_agent_runtime.retrieve(knowledgeBaseId=KB_ID, retrievalQuery={'text':query}, retrievalConfiguration={'vectorSearchConfiguration':{'numberOfResults':max_results}}). Returns slim_result of [{title: location.s3Location.uri split by '/' last part, source_uri, score, snippet: content[:500]}]. Docstring: 'Search the internal Knowledge Base for documents related to a topic. Returns top matching document titles and snippets. Use kb_retrieve for full content.'

2. @mcp.tool() async def kb_retrieve(query: str, num_results: int = 3) → dict: same API call but returns full content [{content, source_uri, page: metadata.get('x-amz-bedrock-kb-chunk-page','unknown'), score}]. Docstring: 'Retrieve full passage content from the Knowledge Base with source citations. Use this after kb_search to get the actual text to cite in your answer.'

Register both via register_kb_tools(mcp). Both tools: try/except format_error, slim_result."

Document Format Tips

FormatSupportedBest practice
PDF✅ BestText-selectable PDFs. Scanned images require Textract first.
Markdown (.md)✅ GreatHeadings help chunking. Use H2/H3 to structure topics.
Plain text (.txt)✅ GoodSimple and reliable. No formatting metadata.
Word (.docx)✅ GoodText extracted. Tables may lose structure.
HTML✅ OKTags stripped. Clean HTML only — not SPA renders.
Images (.png, .jpg)❌ NoUse Textract to extract text first, then upload the .txt
Code files (.py, .ts)⚠️ LimitedWorks but context is lost. Better to document in Markdown.

What You Have Learned

RAG = grounded answers
Without RAG, the AI guesses about your company-specific knowledge. With RAG, it retrieves and cites your actual documents. Citations are the proof — if the AI can't cite it, it's hallucinating.
Bedrock manages all the complexity
Chunking, embedding, vector store, ANN search, reranking — all managed by Bedrock. You upload documents to S3 and call one API. No vector database to maintain.
Two tools for two jobs
kb_search for discovery (which docs exist?), kb_retrieve for reading (what does the doc say?). The AI learns to call search first, then retrieve the most relevant result for citation.
S3 is the source of truth
Upload a new version of a doc to S3 and re-sync — the KB updates automatically. No rebuild, no redeployment. The KB reflects whatever is in S3 after each sync.
