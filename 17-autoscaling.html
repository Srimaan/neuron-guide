Autoscaling

Handle any load without over-provisioning. HPA adds pods when CPU spikes. Cluster Autoscaler adds nodes when pods can't be scheduled. KEDA scales on custom metrics like queue depth. Together they make Neuron elastic and cost-efficient.

⏱ 2–3 hours ☸️ HPA + CA + KEDA 💰 Cost-optimised 🏁 Phase 2 Complete
🎯
Goal: Configure HPA for the agent backend (scale on CPU), Cluster Autoscaler for node scaling, and KEDA for custom metric scaling. Zero manual intervention when load spikes — and cost drops automatically when traffic quiets.
🧱
Deliverable: Under a load test, kubectl get hpa -n neuron-prod shows replicas increasing. When load drops, replicas scale back down within 5 minutes. Cluster Autoscaler adds and removes EC2 nodes automatically.

Three Layers of Kubernetes Scaling

Kubernetes scales at three levels. Each solves a different problem and operates independently. You need all three working together for a fully elastic system.

Figure 17.1 — Three Scaling Layers: HPA, Cluster Autoscaler, KEDA

How HPA Works — The Control Loop

HPA runs a control loop every 15 seconds. It reads CPU metrics from the Metrics Server, compares to the target, and adjusts the replica count. Scale-up is fast (30s). Scale-down is slow (5 min) to avoid flapping.

Figure 17.2 — HPA Control Loop Under Load

Cluster Autoscaler — Node-Level Scaling

When HPA adds so many pods that no node has capacity, pods go Pending. The Cluster Autoscaler detects this and provisions a new EC2 node. When nodes are underutilised for 10 minutes, it terminates them to save cost.

Figure 17.3 — Cluster Autoscaler: Scale-Out and Scale-In Flow

Resource Requests — Why They Matter for Autoscaling

HPA and Cluster Autoscaler both depend on resource requests being set correctly. A pod with no requests is invisible to the autoscaler. An over-requested pod wastes node capacity.

Figure 17.4 — Resource Requests vs Limits: Correct Configuration

Verify Autoscaling Works

bashLoad test and watch HPA respond
# Watch HPA in real time (open in separate terminal)
kubectl get hpa -n neuron-prod -w
# NAME                  REFERENCE              TARGETS    MINPODS  MAXPODS  REPLICAS
# neuron-mcp-client     Deployment/...         23%/70%    3        10       3

# Generate load (replace URL with your ALB address)
for i in {1..50}; do
  curl -s -X POST https://your-alb-url.elb.amazonaws.com/chat \
    -H "Content-Type: application/json" \
    -d '{"prompt":"Analyse all open PRs","session_id":"load-test-'$i'"}' &
done

# Within ~2 minutes, HPA should show replicas increasing:
# neuron-mcp-client  ...  82%/70%  3  10  6  ← scaled up!

# After stopping load — scale down happens after 5 min window
kubectl describe hpa neuron-mcp-client -n neuron-prod
# Events: ScaledUp ... ScaledDown ...

What You Have Learned

HPA + CA work together
HPA adds pods until nodes are full. CA adds nodes when pods go Pending. They operate independently but compose automatically — you configure each once.
Scale-down is slower than scale-up
The 5-minute stabilisation window prevents thrashing — scaling down then up immediately. Scale up fast to handle load, scale down slowly to avoid yo-yo behaviour.
Resource requests are the foundation
Without correct resource requests, neither HPA nor Cluster Autoscaler works properly. Set requests conservatively (what you normally need) and limits at 2× requests.
PDBs protect during scale-in
Without PDBs, Cluster Autoscaler might drain a node that takes all replicas of a service offline. minAvailable:2 means at least 2 copies are always running during any disruption.
Autoscaling halves costs
Fixed 3-node cluster: ~$150/month. Elastic cluster (min 1, scales as needed): ~$80/month. The savings compound — every environment you autoscale reduces cost.
min=1 prevents cold starts
Always keep at least one node warm. Scaling from 0 nodes means waiting 3–4 minutes for EC2 boot on every first request — unacceptable for a team tool.
