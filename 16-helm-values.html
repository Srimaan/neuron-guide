Helm Values &
Multi-Env

One Helm chart, three environments. Use values file layering to run a lean dev cluster at $0, a full-scale staging cluster for testing, and a hardened production cluster for your users — all from the same chart, promoted through the same pipeline.

⏱ 2–3 hours 🌍 3 environments 📋 Values layering 🔐 Secrets management
🎯
Goal: Structure Helm values across three environments (dev, staging, prod) so the same chart deploys correctly everywhere. Add Kubernetes Secrets management for sensitive values. Update the CI/CD pipeline to promote through environments.
🧱
Deliverable: values-dev.yaml, values-staging.yaml, values-prod.yaml. Each environment gets its own namespace. A PR merge deploys to staging automatically. Promotion to prod requires a manual approval step.

How Helm Values Layering Works

Helm merges values files from left to right. Later files override earlier ones. This means values.yaml holds the defaults and environment-specific files only override what differs.

Figure 16.1 — Helm Values Layering: Base + Environment Override

Three Environments — What Differs

Each environment lives in its own Kubernetes namespace and has different resource allocations. Dev is lean to keep costs low. Staging matches production to catch resource-related bugs before they hit users.

Figure 16.2 — Environment Comparison: Dev vs Staging vs Prod

Environment Promotion Pipeline

Code moves through environments in one direction only. Automatic for dev and staging. Manual approval gate before production. If staging looks bad, you never promote — production is never affected.

Figure 16.3 — Environment Promotion Flow

Secrets Management — What NOT to Put in values.yaml

Values files are checked into Git. Never put sensitive values there. We use Kubernetes Secrets for credentials, with different approaches for each environment.

Figure 16.4 — What Goes in values.yaml vs Kubernetes Secrets

Each file also sets namespace: neuron-{env}. Show only the values that differ from base, not the full file."
Step 16.2 — Create Integration Secrets
Store GitHub, Jira, Slack tokens as Kubernetes Secrets
bashCreate secrets in each namespace
# Create namespace first
kubectl create namespace neuron-staging
kubectl create namespace neuron-prod

# Create integration credentials secret (do this per environment)
kubectl create secret generic neuron-integrations \
  --from-literal=GITHUB_TOKEN=ghp_xxx \
  --from-literal=JIRA_API_TOKEN=xxx \
  --from-literal=JIRA_DOMAIN=yourco.atlassian.net \
  --from-literal=JIRA_USER=you@yourco.com \
  --from-literal=SLACK_BOT_TOKEN=xoxb-xxx \
  --from-literal=MCP_AUTH_TOKEN=your-random-token \
  -n neuron-staging

# Repeat for neuron-prod with production credentials
# (can use different Jira projects / Slack workspaces per env)

Step 16.4 — Update deploy.yml for Multi-Environment
Auto-deploy to staging on main, manual approval gate for prod
✨
Claude Prompt:

"Update .github/workflows/deploy.yml to support multi-environment deployment.

After the existing build+push steps add two deployment jobs:

deploy-staging (runs after build, no approval): helm upgrade --install neuron ./helm/neuron -n neuron-staging --create-namespace -f helm/neuron/values.yaml -f helm/neuron/values-staging.yaml --set image.tag={IMAGE_TAG} --wait --timeout 5m. Post success/failure to GitHub Step Summary.

deploy-prod (needs: deploy-staging, requires manual approval via GitHub environment protection): same helm command but -n neuron-prod -f values-prod.yaml. Use environment: name: production in the job to trigger the approval gate. Show how to set up the GitHub environment protection rule requiring a reviewer."

Verify Multi-Environment Works

bashCheck both environments are running
# After a main merge triggers the pipeline:

# Check staging deployed
kubectl get pods -n neuron-staging
# 2x mcp-server, 2x mcp-client, 1x frontend — all Running

# Check staging uses the right model (Sonnet 4)
kubectl get deployment neuron-mcp-client -n neuron-staging \
  -o jsonpath='{.spec.template.spec.containers[0].env}'

# Approve prod deployment in GitHub Actions UI
# Then verify prod:
kubectl get pods -n neuron-prod
# 3x mcp-server, 3x mcp-client, 2x frontend — all Running

# Dry-run to preview what would change before deploying
helm upgrade --install neuron ./helm/neuron \
  -n neuron-prod \
  -f helm/neuron/values.yaml \
  -f helm/neuron/values-prod.yaml \
  --set image.tag=abc1234 \
  --dry-run

Essential Helm Commands

CommandWhat it doesWhen to use
helm diff upgradeShows what would change before deployingBefore every production deploy
helm history neuron -n neuron-prodLists all previous releases with revision numbersAuditing, pre-rollback check
helm rollback neuron 3 -n neuron-prodRolls back to revision 3Emergency — bad deploy in prod
helm get values neuron -n neuron-prodShows currently deployed valuesDebugging what is actually running
helm template neuron ./helm/neuron -f values-prod.yamlRenders final YAML without deployingPreviewing the rendered manifests

Checkpoint

✅ All of these must be true:

  1. Three values files exist: values-dev.yaml, values-staging.yaml, values-prod.yaml
  2. Merging to main auto-deploys to neuron-staging namespace
  3. Production deploy requires a manual approval in GitHub Actions
  4. Integration credentials are in a Kubernetes Secret, not in values files
  5. kubectl get pods -n neuron-staging shows correct replica counts (2×2×1)
  6. helm rollback successfully reverts a staging deployment

What You Have Learned

Values layering keeps DRY
Base values.yaml holds 90% of config. Environment files only override what differs. One change in the base applies to all environments — no copy-paste drift between envs.
Staging must mirror prod config
Staging uses the same Bedrock model, same resource ratios, same K8s config as prod — just fewer replicas. If staging passes, prod will pass. Different configs defeat the purpose.
Manual approval gate = safety
Automated staging + manual prod approval is the sweet spot. You get the speed of automation without the risk of automatic production deploys after every commit.
Secrets never touch Git
API tokens and passwords live in Kubernetes Secrets, created once via kubectl and never stored anywhere else. Rotation means updating one Secret — no code changes, no redeploys.
