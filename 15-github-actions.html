GitHub Actions
CI/CD

Automate every deployment. When code merges to main, GitHub Actions runs tests, builds Docker images, pushes to ECR, and deploys the new version to EKS — all without human intervention. Push to deploy in under 8 minutes.

⏱ 3–4 hours 💰 Free (GitHub Actions) 🔄 Push to deploy ⏱ ~8 min pipeline
🎯
Goal: A single git push origin main triggers a pipeline that tests, builds, pushes, and deploys Neuron to EKS automatically. Zero manual steps after code merge.
🧱
Deliverable: Two GitHub Actions workflows — ci.yml (runs on every PR: lint + test) and deploy.yml (runs on main merge: build + push ECR + Helm deploy to EKS). A green pipeline badge on your README.

The Complete CI/CD Pipeline

Two separate workflows with distinct responsibilities. The CI workflow runs on every pull request to catch issues early. The deploy workflow runs only on merges to main to push to production.

Authenticating GitHub Actions to AWS — No Secrets

We use GitHub OIDC to let GitHub Actions authenticate to AWS without storing any AWS credentials in GitHub Secrets. Same principle as IRSA — temporary tokens, nothing to rotate.

Create the IAM role that GitHub Actions will assume via OIDC

Create IAM role 'neuron-github-actions-role' with policies: AmazonECR_FullAccess, and an inline policy for EKS access (eks:DescribeCluster and eks:AccessKubernetesApi). Also show the AWS CLI commands to: create the OIDC provider, create the role, and attach the policies."
Step 15.2 — Write the CI Workflow
.github/workflows/ci.yml — runs on every PR
✨
Claude Prompt:

"Generate .github/workflows/ci.yml for the Neuron monorepo. Trigger: pull_request to main. Run 3 parallel jobs — one for each service (neuron-mcp-server, neuron-mcp-client, neuron-frontend) — each job: (1) checkout code, (2) set up Python 3.11 and uv, (3) install deps with 'uv sync --frozen', (4) run ruff check, (5) run pytest tests/ -v with continue-on-error: false. Use ubuntu-latest runner. Cache uv virtualenv using actions/cache with key based on uv.lock hash. Report job status in PR checks."
Step 15.3 — Write the Deploy Workflow
.github/workflows/deploy.yml — runs on merge to main
✨
Claude Prompt:

"Generate .github/workflows/deploy.yml for Neuron. Trigger: push to main. Needs: permissions id-token: write, contents: read (for OIDC). Steps:
1. Checkout code
2. Configure AWS credentials using aws-actions/configure-aws-credentials@v4 with role-to-assume: ARN of neuron-github-actions-role, aws-region: us-east-1 — NO stored secrets
3. Login to ECR with aws-actions/amazon-ecr-login@v2
4. Set IMAGE_TAG env var to github.sha (short: 7 chars)
5. Build and push 3 Docker images in parallel using matrix strategy: services: [neuron-mcp-server, neuron-mcp-client, neuron-frontend]. Each: docker build -t {ECR_REGISTRY}/{service}:{IMAGE_TAG}, docker push
6. Update kubeconfig: aws eks update-kubeconfig --name neuron-cluster --region us-east-1
7. Helm upgrade: helm upgrade --install neuron ./helm/neuron --namespace neuron --set image.tag={IMAGE_TAG} --set image.registry={ECR_REGISTRY} --wait --timeout 5m0s
8. Post deployment summary to GitHub Step Summary showing which images were pushed and the image tag

Use env vars: AWS_ACCOUNT_ID from GitHub Variables (not secrets), ECR_REGISTRY built from account ID."
Step 15.4 — Protect the Main Branch
Require CI to pass before any merge to main
  1. Go to GitHub repo → Settings → Branches
  2. Click Add branch protection rule
  3. Branch name pattern: main
  4. Check Require status checks to pass before merging
  5. Search for and add: test (neuron-mcp-server), test (neuron-mcp-client), test (neuron-frontend)
  6. Check Require branches to be up to date before merging
  7. Save — now no merge can happen without green CI

Checkpoint

✅ All of these must be true:

  1. Opening a PR triggers ci.yml — 3 parallel test jobs run and show in the PR checks
  2. A test failure blocks the merge (red status check)
  3. Merging to main triggers deploy.yml automatically
  4. Deploy workflow uses OIDC — zero AWS credentials in GitHub Secrets
  5. After pipeline completes, kubectl get pods -n neuron shows updated image SHAs
  6. The GitHub Actions tab shows a green workflow run with all steps passed

What You Have Learned

CI and CD are separate concerns
CI runs on every PR — fast, cheap, gives feedback early. CD runs on main merge — slower, deploys to production. Separating them lets you deploy frequently without CI noise.
OIDC eliminates secret sprawl
No AWS credentials in GitHub Secrets means nothing to rotate, nothing to leak, nothing to audit. GitHub → AWS via OIDC is the same zero-credential pattern as IRSA in pods.
Git SHA tags are immutable
Tagging images with git SHA creates a permanent, auditable link between a deployment and a code commit. You can always see exactly what code is running and roll back to any previous commit.
Branch protection enforces quality
Requiring CI checks before merge is not bureaucracy — it's the minimum viable quality gate. One flaky test that passes manually but fails in CI catches more bugs than most code reviews.
