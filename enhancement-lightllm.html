LightLLM
Model Router

Add a self-hosted open-weight model alongside Bedrock. Route cheap utility tasks (summarisation, classification, code review) to your own GPU endpoint. Keep Claude Sonnet 4 for everything that needs it. Cut utility token costs by up to 95%.

💰 Up to 95% cheaper for utility tasks 🔀 Hot-swap between models 🏗 Non-breaking change GPU EC2 or EKS node

Applies to chapters

Ch 5 — Agent Backend Ch 13 — Session Memory (summarisation) Ch 18 — Multi-Agent (routing model) All future chapters using AI calls
🎯
Goal: Add a ModelRouter class that selects between Bedrock and LightLLM based on task type. Every AI call in Neuron goes through the router. Model switching requires one config change — zero code changes.

Task Routing — What Goes Where

Not every AI call needs Claude Sonnet 4. The expensive model earns its cost on complex reasoning and user-facing responses. Utility tasks — summarisation, classification, batch processing — can run on a self-hosted 70B model at a fraction of the cost.

Build Steps

Step L1 — Add LightLLM to Docker Compose (Local Dev)
Use Ollama for CPU-based local testing — no GPU required
yamlAdd to docker-compose.yml
# docker-compose.yml addition
services:
  lightllm:
    image: ollama/ollama:latest   # CPU mode for local dev
    ports:
      - "8080:11434"
    volumes:
      - ollama-models:/root/.ollama  # cache downloaded models
    networks:
      - neuron-network
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:11434"]
      interval: 30s

volumes:
  ollama-models:   # persist model weights across restarts
bashPull the model once (run after make up)
# Pull Llama 3.2 8B for local testing (3.8GB download)
docker exec -it neuron-lightllm ollama pull llama3.2:8b

# Or pull Qwen 2.5 7B — better quality, similar size
docker exec -it neuron-lightllm ollama pull qwen2.5:7b

# Verify the OpenAI-compat endpoint works
curl http://localhost:8080/api/chat \
  -d '{"model":"qwen2.5:7b","messages":[{"role":"user","content":"hello"}]}'
Step L2 — Create model_router.py
The routing brain — picks provider based on task type and config
✨
Claude Prompt:

"Create mcp_client/services/model_router.py for Neuron. This class selects between Bedrock and LightLLM based on task type.

Config (from Settings): MODEL_PROVIDER (str, 'auto' | 'bedrock' | 'lightllm'), LIGHTLLM_URL (str, 'http://lightllm:8080'), LIGHTLLM_MODEL (str, 'qwen2.5:7b'), UTILITY_TASKS (list[str], ['summarise','classify','code_review','batch']).

class ModelRouter:
- UTILITY_TASKS: set of task types that can use LightLLM
- get_bedrock_client(model_id: str) → BedrockClient: returns configured boto3 bedrock-runtime client
- get_lightllm_client() → OpenAI: returns openai.OpenAI(base_url=settings.lightllm_url+'/v1', api_key='ollama') using the openai SDK pointing at local endpoint
- async get_client(task_type: str) → tuple[client, model_id]: if MODEL_PROVIDER=='bedrock' always returns Bedrock. If 'lightllm' always returns LightLLM. If 'auto': UTILITY_TASKS tasks get LightLLM, others get Bedrock. If LightLLM health check fails, fall back to Bedrock and log warning.
- async health_check_lightllm() → bool: GET {lightllm_url}/api/tags timeout 2s.

Singleton pattern: cache the router in app.state.model_router. Full type hints."

Step L5 — Deploy LightLLM on EKS GPU Node (Production)
Add a GPU node group and deploy vLLM with a real 70B model for staging/prod
✨
Claude Prompt:

"Generate the EKS GPU node group config and Helm chart for LightLLM in production.

1. eksctl nodegroup config: name 'gpu-nodes', instance type g4dn.2xlarge (1× T4 GPU), min 0 max 2, spot instances for cost saving, labels gpu=true, taints nvidia.com/gpu=true:NoSchedule
2. helm/neuron/templates/lightllm.yaml: Deployment with nvidia.com/gpu: 1 resource request, image vllm/vllm-openai:latest, args: --model Qwen/Qwen2.5-72B-Instruct-AWQ --quantization awq --max-model-len 8192, tolerations for GPU taint, nodeSelector gpu=true, Service ClusterIP port 8080
3. values.yaml addition: lightllm.enabled: false (dev), lightllm.enabled: true (prod), lightllm.model: 'Qwen/Qwen2.5-72B-Instruct-AWQ'
Note: AWQ quantization allows 72B model to fit in 24GB GPU VRAM."

Verify the Router Works

bashTest routing behaviour
# Start with LightLLM running (after make up + model pull)

# Check health endpoint shows LightLLM status
curl http://localhost:8000/health
# {"lightllm":"connected","lightllm_model":"qwen2.5:7b", ...}

# Trigger a session rotation — should use LightLLM for summary
# Watch logs for: "Summarising via lightllm: qwen2.5:7b"
docker-compose logs neuron-mcp-client | grep "Summarising via"

# Test fallback: stop LightLLM, make a request
docker-compose stop lightllm
# Should see: "LightLLM unreachable, falling back to Bedrock"

# Switch entire platform to Bedrock-only (e.g. LightLLM outage)
# Just set in .env:
echo "MODEL_PROVIDER=bedrock" >> .env && make restart

Checkpoint

✅ All of these must be true:

  1. LightLLM container starts and /api/tags returns the pulled model
  2. Session summarisation logs show "Summarising via lightllm" not Bedrock
  3. Intent classification logs show LightLLM provider for routing calls
  4. User-facing chat responses still go to Bedrock (check logs for "via bedrock")
  5. Stopping LightLLM container — all tasks fall back to Bedrock gracefully, no errors
  6. MODEL_PROVIDER=bedrock forces all tasks to Bedrock regardless of task type
💡
Start small: Begin with qwen2.5:7b (CPU, no GPU needed) in development. Once you're confident in the routing logic, switch to Qwen2.5-72B-Instruct-AWQ on a GPU node in production. The code changes nothing — only the model name in config.
