Response
Limiting

Add three layers of protection against context overflow. A single Jira search returning 500 tickets can silently break the entire conversation. We fix this with pre-serialization capping, a hard middleware limit, and chunked retrieval for legitimately large data.

⏱ 2–3 hours 💰 $0 🛡 Defence in depth 3 protection layers
🔴
This is not optional. Without response limiting, one search returning large data silently overflows Claude's context window. The AI starts hallucinating, forgetting previous messages, or refusing to respond. You won't always see an obvious error — the conversation just gets progressively worse.
🧱
Deliverable: Enhanced slim_result() handling nested structures, a ResponseLimitingMiddleware hard-capping at 75KB, and a get_data_chunk() tool for progressive retrieval of large data.

Three Layers of Protection

We use defence in depth — three layers, each catching overflow that slipped through the previous one. Layer 1 handles 95% of cases. Layer 2 handles edge cases. Layer 3 handles legitimate large data.

Figure 12.1 — Three-Layer Response Limiting Architecture

Layer 1 — Enhanced slim_result()

We already have a basic slim_result() from Chapter 3. Now we enhance it to handle nested structures — dictionaries of lists of dictionaries with long string values — that the basic version misses.

Figure 12.2 — What slim_result() Does to Nested Data

Layer 3 — Chunked Retrieval for Large Data

Sometimes data is legitimately large — a 50KB build log, a full file, a complete test suite output. We cannot truncate it without losing meaning. Instead we store it and let the AI retrieve it progressively in 5KB chunks.

✅ All of these must be true:

  1. slim_result([...500 items...]) returns exactly 50 items with _truncated: True
  2. slim_result({"desc": "x" * 10000}) returns description truncated to 5000 chars
  3. slim_result({"items": [{...long strings...}, ...500 items]}) caps list AND truncates strings
  4. get_data_chunk(data_id, chunk_index=0) returns correct chunk content and total_chunks
  5. Middleware is registered and catches responses over 75KB

What You Have Learned

Context overflow fails silently
When Claude's context window is full, it doesn't throw an error — it starts hallucinating or truncating its own response. Response limiting prevents this from ever happening.
Defence in depth
Three layers because one is not enough. slim_result catches most cases. Middleware catches edge cases. Chunked storage handles legitimate large data without truncating meaning.
The _truncated flag matters
When slim_result truncates data it sets _truncated: True. The AI reads this and knows to tell the user "I only showed you the first 50 results, there are 450 more."
TTL prevents memory leaks
The in-memory store expires data after 1 hour and caps at 100 items. Without TTL, every large response accumulates in memory until the server runs out of RAM.
