How Claude Desktop Uses MCP

Claude Desktop can connect to MCP servers in two ways. We use stdio in this chapter because it requires zero network configuration — Claude Desktop just launches your Python script directly.

Setup Steps

Step 4.1 — Install Claude Desktop
Download and install the Claude Desktop application
  1. Go to claude.ai/download
  2. Download the version for your operating system
  3. Install and sign in with your Anthropic account
  4. You need Claude Pro ($20/month) or higher to use MCP servers
⚠️
No Claude Pro? You can skip this chapter and come back later. Chapter 5 builds the AI backend that powers the web UI — no Claude Desktop needed. Every feature works through the web app. Claude Desktop is just a fast way to prove the MCP server works.
Step 4.2 — Find the Config File Location
Claude Desktop reads a JSON config file to discover MCP servers

📁 Config File Location by Operating System

macOS ~/Library/Application Support/Claude/claude_desktop_config.json
Windows %APPDATA%\Claude\claude_desktop_config.json
Linux ~/.config/Claude/claude_desktop_config.json

If the file doesn't exist yet, create it. Claude Desktop creates the directory on first launch.

Step 4.3 — Write the Config File
Tell Claude Desktop where your MCP server is and how to launch it

Open the config file in any text editor. Paste in this JSON — replacing the path and credentials with your real values.

jsonclaude_desktop_config.json
{
  "mcpServers": {
    "neuron": {
      "command": "uv",
      "args": [
        "--directory",
        "/FULL/PATH/TO/neuron-platform/neuron-mcp-server",
        "run",
        "python",
        "-m",
        "mcp_server.main"
      ],
      "env": {
        "TRANSPORT":      "stdio",
        "GITHUB_TOKEN":   "ghp_your_token_here",
        "JIRA_DOMAIN":    "yourcompany.atlassian.net",
        "JIRA_USER":     "your@email.com",
        "JIRA_API_TOKEN": "your_jira_api_token",
        "SLACK_BOT_TOKEN":"xoxb-your-slack-token"
      }
    }
  }
}
🔴
Replace the path! Change /FULL/PATH/TO/neuron-platform/neuron-mcp-server to the actual path on your laptop. Run cd neuron-mcp-server && pwd in your terminal to get it. It must be the absolute path — relative paths do not work.
Step 4.4 — Restart Claude Desktop
Close completely and reopen to load the new config
  1. macOS: Right-click the dock icon → Quit. Do not just close the window.
  2. Windows: Right-click the system tray icon → Exit
  3. Reopen Claude Desktop
  4. Start a new conversation
  5. Look for the hammer icon (🔨) at the bottom of the input box
  6. Click it — you should see your 5 Neuron tools listed
Figure 4.2 — What You Should See in Claude Desktop After Connecting
🔧
Don't see the tools? Open Claude Desktop logs:

macOS: Help menu → Open Logs Folder → check mcp*.log
Windows: %APPDATA%\Claude\logs\

Common fixes: wrong absolute path in config · invalid JSON syntax · Python or uv not found (use full path like /usr/local/bin/uv) · .env file not in the right folder

What You Have Learned

stdio transport — no config needed
Claude Desktop spawns your Python script as a child process. No ports, no auth tokens, no network config. Communication happens over stdin/stdout. Perfect for local development.
The AI chains tools automatically
You did not tell Claude to call GitHub first then Jira. It planned the sequence itself. This is the "agentic" part — reasoning through multi-step tasks without explicit instruction.
One MCP server, many clients
The same neuron-mcp-server works for Claude Desktop now and for your web app in Chapter 5. You don't write the tools twice — one server, any number of clients.
Real actions, not simulation
The Jira ticket created in Test 2 is a real ticket on your board. The GitHub API was called with your real credentials. This is not a sandbox — treat it carefully.
config.json controls everything
Adding a new MCP server to Claude Desktop is just adding a new entry in the JSON file. No Claude Desktop code changes. The config file is the entire integration surface.
Never write to stdout in stdio mode
If you print() anything in your server when running in stdio mode, it corrupts the JSON-RPC protocol and breaks the connection. Always use stderr for logs.
