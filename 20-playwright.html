Browser Agent

Give the AI eyes and hands on the web. Navigate URLs, take screenshots, click elements, fill forms, extract page content, and monitor live dashboards — all triggered by plain-English requests. The Browser Agent from Chapter 18 gets its tools here.

⏱ 3–4 hours 🎭 Playwright 📸 Screenshots 🔐 Domain allowlist
🎯
Goal: Add a containerised Playwright browser to the stack and expose 8 browser tools through the MCP server. The AI can navigate, screenshot, click, extract text, and monitor web pages — with a strict domain allowlist for safety.
🧱
Deliverable: "Screenshot our staging dashboard and check if the health page shows any errors" works end-to-end. The AI navigates to the URL, takes a screenshot, reads the DOM, and reports back — all in under 10 seconds.

What the Browser Agent Can Do

The AI gets a headless Chrome browser. It can see pages (screenshots + DOM text), interact with them (click, fill, submit), and extract structured data — all tools that were previously only possible with manual human steps.

Architecture — Containerised Browser

The browser runs in its own Docker container, isolated from the other services. The MCP server connects to it over a REST API. This isolation means a broken page can't crash the MCP server.

Figure 20.2 — Playwright Container Architecture and Tool Flow

The 8 Browser Tools

browser_navigateNavigate to a URL and wait for DOM content loaded. Returns page title and current URL. Enforces domain allowlist.
browser_screenshotTake a full-page or viewport screenshot. Returns base64 PNG. The AI uses Claude's vision to describe what it sees.
browser_get_textExtract all visible text from the page (no HTML tags). Optionally scope to a CSS selector. Returns clean text, slim_result at 8000 chars.
browser_get_htmlGet raw HTML of a selector. For extracting structured data (tables, lists) the AI can parse. slim_result at 5000 chars.
browser_clickClick an element by CSS selector, text content, or aria-label. Waits for the element to be visible before clicking.
browser_fillType text into an input field by CSS selector. Clears existing content first. Supports text inputs, textareas, search boxes.
browser_wait_forWait up to N seconds for a CSS selector to appear or text to be visible on the page. Returns true/false. Use for polling deploys.
browser_evaluateExecute JavaScript in the page context and return the result. For extracting data that isn't visible in the DOM.

Security — The Domain Allowlist

Giving an AI access to a browser is powerful and risky. We use a strict domain allowlist — the browser will not navigate to any URL not on the list. This prevents prompt injection attacks from external pages redirecting the AI to malicious sites.

Build Steps

Step 20.1 — Create the Playwright Service
FastAPI microservice wrapping Playwright — runs in its own container
✨
Claude Prompt:

"Create neuron-playwright/ as a new Python service. Tech stack: Python 3.11, FastAPI, playwright (async API), uv.

Create main.py with a FastAPI app and a persistent browser context (single Chromium instance reused across requests for speed). Endpoints:
POST /navigate: {url: str} → {title, url, status_code} — checks allowlist first
POST /screenshot: {full_page: bool=True} → {image_base64: str, width, height}
POST /get_text: {selector: str=''} → {text: str} — if selector empty get all visible text
POST /get_html: {selector: str} → {html: str}
POST /click: {selector: str} → {success: bool, element_found: bool}
POST /fill: {selector: str, value: str} → {success: bool}
POST /wait_for: {selector: str='', text: str='', timeout_ms: int=5000} → {found: bool, elapsed_ms: int}
POST /evaluate: {script: str} → {result: any}
GET /health → {status: ok, browser: running}

Allowlist: read BROWSER_ALLOWED_DOMAINS env var (comma-separated). Check domain before any navigate call. Wildcard subdomain support (*.myapp.com matches staging.myapp.com). Return 403 with {error: 'Domain not allowed'} if blocked.

Browser options: headless=True, --no-sandbox, --disable-gpu, viewport 1280×800. Use async_playwright context manager stored in app.state."
Step 20.2 — Add Dockerfile for Playwright Service
Playwright needs Chromium system deps — use the official image
dockerfileneuron-playwright/Dockerfile
# Use Microsoft's Playwright base image — has all Chromium deps
FROM mcr.microsoft.com/playwright/python:v1.48.0-jammy

WORKDIR /app

# Install uv
RUN pip install uv --break-system-packages

# Layer cache: deps before code
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev

COPY . .

# Non-root user
RUN useradd -m -u 1000 appuser && chown -R appuser /app
USER appuser

EXPOSE 3000
HEALTHCHECK CMD curl -f http://localhost:3000/health
CMD ["uv", "run", "uvicorn", "main:app", "--host", "0.0.0.0", "--port", "3000"]

Step 20.4 — Add Browser Tools to MCP Server
8 tools in browser_tools.py — proxy calls to the Playwright service
✨
Claude Prompt:

"Create mcp_server/tools/browser_tools.py for neuron-mcp-server. Settings: PLAYWRIGHT_URL (str, 'http://neuron-playwright:3000'). Uses httpx to call the Playwright service.

Create 8 @mcp.tool() async functions, each proxying to the matching Playwright service endpoint. Each tool: validates args, calls httpx.AsyncClient().post(f'{PLAYWRIGHT_URL}/{endpoint}', json=payload, timeout=30), slim_result on responses, format_error on exceptions.

Special handling for browser_screenshot: response contains image_base64 field. Return {image_base64, width, height, note: 'Pass this to Claude vision to describe what is visible'}.

Special handling for browser_get_text: slim_result truncate at 8000 chars with _truncated flag.

Register via register_browser_tools(mcp). Include list_browser_ops() and browser_tool() dispatcher following Chapter 11 pattern."

Configuring the Domain Allowlist

bashSet allowed domains in .env
# neuron-mcp-server/.env — add these lines
PLAYWRIGHT_URL=http://neuron-playwright:3000

# neuron-playwright container environment (in docker-compose.yml)
# Comma-separated. Wildcards: *.domain.com matches all subdomains
BROWSER_ALLOWED_DOMAINS=*.mycompany.com,staging.myapp.com,localhost:*

# Examples of what gets blocked (not on list):
# google.com, github.com, any external site

What You Have Learned

Browser as a tool, not a product
The AI doesn't "use a browser" — it calls discrete tools (navigate, screenshot, click). Each tool has clear inputs and outputs. The AI sequences them like any other tools.
Vision + browser = understanding
A screenshot returned as base64 can be passed to Claude's vision capability. The AI can describe what it sees — errors, layouts, form states — without any DOM parsing.
Isolation prevents blast radius
The Playwright container is isolated — it can't reach internal services. If the AI is tricked into navigating to a malicious page, that page can't access your database or APIs.
Domain allowlist is non-negotiable
Without an allowlist, prompt injection via external pages could direct the AI to hostile sites. The allowlist must be the first check in every navigate call — before Playwright ever opens a tab.
