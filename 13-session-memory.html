The Memory Architecture

Claude has no built-in memory between conversations. We build it using DynamoDB as the storage layer and a three-tier memory model: recent messages, rolling summary, and archived sessions.

Figure 13.1 — Three-Tier Session Memory Model

Session Rotation at 50 Messages

When a session hits 50 messages we don't delete history — we rotate it. Claude Haiku summarises the session in under 1 second, the summary is stored, and a fresh session starts with that summary as context.

Figure 13.2 — Session Rotation Flow

Multi-User Session Isolation

Each user gets completely isolated session history. The session ID is a combination of user_id + session_token — different users never see each other's conversations.

Figure 13.3 — Multi-User Session Isolation in DynamoDB

"Enhance mcp_client/services/session.py SessionManager with session rotation and summarisation.

Add to existing class:
1. ROTATION_THRESHOLD = 50 (class constant)
2. async _generate_summary(messages: list[dict]) → str — calls Bedrock with Claude Haiku (use settings.bedrock_model_id_haiku, default 'anthropic.claude-haiku-4-20240307-v1:0'), prompt: 'Summarise this conversation history in 2-3 paragraphs. Focus on: decisions made, tasks completed, key context that would help continue the conversation. Be concise.' Return summary string or '' on error.
3. async _archive_session(session_id: str, messages: list) → None — save to DynamoDB with key f'{session_id}:archive:{timestamp}', TTL 30 days
4. async should_rotate(session_id: str) → bool — returns True if message count >= ROTATION_THRESHOLD
5. async rotate_session(session_id: str) → str — generates summary, archives current session, creates new session with f'{session_id}:v{n}' ID, copies summary field. Returns new session_id.
6. Modify save_history() — after saving, if len(messages) >= ROTATION_THRESHOLD, call rotate_session() and return new session_id. Update the caller (chat.py router) to use returned session_id.
7. Modify get_history() — also return summary field. Caller should prepend summary as a system message if present.

Multi-user: session_id should be '{user_id}:{uuid}'. Add user_id parameter to get_history and save_history."

What You Have Learned

LLMs have no innate memory
Claude starts fresh on every request. All "memory" is context we inject. Building good memory = deciding what to store and how much to inject without overflowing the window.
Summarisation preserves meaning cheaply
50 messages of conversation compresses to ~300 tokens via summarisation with minimal loss. This makes unlimited conversation length economically viable.
Use Haiku for utility tasks
Summarisation doesn't need Claude Sonnet 4's full intelligence. Haiku is 12× cheaper. Use Sonnet for reasoning, Haiku for utility tasks like summarisation and classification.
TTL is your garbage collector
DynamoDB TTL auto-deletes expired sessions — no cron job, no cleanup script. Set aggressive TTLs (24h for active, 30d for archives) and let DynamoDB manage storage.
