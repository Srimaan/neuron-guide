How FastMCP Works

FastMCP is a Python framework that turns regular Python functions into MCP tools with a single decorator. It reads your type hints and docstring automatically — no manual schema writing needed.

Figure 3.1 — How @mcp.tool() Turns a Function Into an AI Tool
💡
Why this matters: The AI reads the "description" field to understand when to call this tool. A good docstring is not optional — it is the instruction manual the AI reads before deciding whether to call your tool. Write it as if you are explaining the tool to a new team member.

The 5 Tools We Build

Each tool is a Python async function decorated with @mcp.tool(). Together they give the AI the ability to search code, manage tickets, and communicate with your team.

github_search_prs()
Searches GitHub pull requests by query string, repo, and state (open/closed/all)
GitHub API
github_get_pr_diff()
Gets the complete diff of a PR — all files changed, additions, deletions, patches
GitHub API
jira_search_issues()
Searches Jira using JQL query language — find tickets by assignee, status, sprint, label
Jira API
jira_create_issue()
Creates a new Jira issue with summary, description, type, priority, and assignee
Jira API
slack_send_message()
Posts a message to any Slack channel or thread, using channel name or ID
Slack API

The slim_result() Utility — Non-Negotiable

Before any tool returns data to the AI, it must pass through slim_result(). This function protects the AI's context window from overflow. Without it, a single Jira search returning 500 tickets would break the entire conversation.

What You Have Learned

FastMCP — one decorator
@mcp.tool() on any async Python function automatically creates a fully-schema'd MCP tool. Type hints become the parameter schema. Docstring becomes the description.
Docstrings are instructions
The AI reads your docstring to decide when to call the tool. A vague docstring produces wrong tool calls. A clear docstring produces perfect tool calls. Write them well.
slim_result() is non-negotiable
Context overflow is a silent killer. 500 Jira tickets = 150K tokens = broken conversation. Apply slim_result() on every list and large dict before returning to the AI.
Tools never raise exceptions
A tool that throws an exception crashes the entire tool call cycle. Always catch exceptions and return format_error(). Let the AI handle the error gracefully.
register_*_tools() pattern
Each category registers itself via a function. main.py stays clean — it just calls register_github_tools(mcp). Adding a new category is one function call.
Two transports, one server
stdio for Claude Desktop (subprocess), SSE HTTP for everything else (network). The TRANSPORT env var switches between them. Same server, same tools, different delivery.
