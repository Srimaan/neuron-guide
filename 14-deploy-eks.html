Deploy to
EKS

Take the Docker containers you built in Chapter 7 and run them on Amazon EKS — AWS's managed Kubernetes service. No credential files in containers, auto-scaling, and health checks that restart failed pods automatically.

⏱ 4–6 hours 💰 ~$80/month ☸️ Kubernetes 🔐 IRSA
🎯
Goal: Create an EKS cluster, push Docker images to ECR, deploy all three Neuron services with Helm, and configure IRSA so pods can call Bedrock and DynamoDB with no credentials stored anywhere.
🧱
Deliverable: Neuron running on EKS accessible via a public Load Balancer URL. kubectl get pods shows all pods Running. IRSA configured — zero AWS credentials in any container image or environment variable.

What Changes in Phase 2

Figure 14.1 — Phase 1 (Laptop) vs Phase 2 (AWS EKS)

EKS Architecture Overview

EKS manages the Kubernetes control plane for you. You create worker nodes (EC2 instances) and deploy your containers as Pods. A Load Balancer exposes the frontend to the internet. IRSA gives pods AWS permissions without any credentials.

Figure 14.2 — EKS Cluster Architecture

IRSA — No Credentials in Pods

IRSA (IAM Roles for Service Accounts) lets Kubernetes pods assume an AWS IAM role automatically — no AWS_ACCESS_KEY_ID, no .env files, no secrets to rotate. The pod gets temporary credentials that auto-expire every hour.

Figure 14.3 — How IRSA Works: Pod → IAM Role → AWS API

Helm Charts — Packaging the Deployment

Helm is the package manager for Kubernetes. Instead of writing raw YAML for every resource, you write a Helm chart once and deploy it to any environment (dev, staging, prod) with different values.

Figure 14.4 — Helm Chart Structure for Neuron

Phase 2 Cost Breakdown

Running a 2-node EKS cluster costs around $80/month. This is the minimum viable production setup. The main costs are EC2 worker nodes — all other services are pay-per-use.

EKS cluster fee$0.10/hr → ~$73/mo
2× t3.medium worker nodes (24×7)~$60/mo total
ALB Load Balancer~$16/mo
ECR image storage (3 images)~$1/mo
DynamoDB (real AWS, low traffic)~$0.25/mo
Bedrock Claude Sonnet 4Pay per token used
💡
Save money: Use Spot Instances for worker nodes (up to 70% cheaper). Add a cluster autoscaler that scales to 0 nodes at night. Schedule node groups to shutdown on weekends. A well-optimised dev cluster can run for ~$25/month.

# kubectl — manages Kubernetes resources

# helm — deploys Helm charts

Step 14.5 — Write and Deploy Helm Charts
Package Neuron as a Helm chart and deploy to the cluster
✨
Claude Prompt:

"Generate a complete Helm chart for Neuron at helm/neuron/. Include:
Chart.yaml: name neuron, version 0.1.0, appVersion 1.0.0
values.yaml: image registry (ECR URL), image tags (latest), replicaCount 1 for each service, resources: mcp-server 256Mi/250m, mcp-client 512Mi/500m, frontend 256Mi/250m, serviceAccount name neuron-agent-sa, namespace neuron, env vars for MCP_SERVER_URL (http://neuron-mcp-server:8001/mcp) and BACKEND_URL (http://neuron-mcp-client:8000)

bashDeploy and verify
# Install ALB Ingress Controller (needed for Load Balancer)
helm repo add eks https://aws.github.io/eks-charts
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \

  --set serviceAccount.create=false \

Checkpoint

✅ All of these must be true:

  1. kubectl get nodes shows 2 nodes in Ready state
  2. kubectl get pods -n neuron shows all 3 pods Running, 0 restarts
  3. The ALB URL opens Neuron's Streamlit UI in the browser
  4. Sending a chat message works end-to-end (Bedrock is reachable via IRSA)
  5. kubectl describe pod [mcp-client-pod] -n neuron shows the IRSA service account annotation
  6. No AWS_ACCESS_KEY_ID or AWS_SECRET_ACCESS_KEY in any pod env vars

What You Have Learned

EKS manages the control plane
You pay for and manage worker nodes. AWS manages the Kubernetes API server, etcd, scheduler, and controller manager. No Kubernetes expertise needed for the hard parts.
IRSA eliminates credential management
No .env files, no Kubernetes Secrets for AWS creds, no rotation schedules. Pods get temporary credentials via OIDC federation. This is the production standard — use it from day one.
Helm = reproducible deployments
One Helm chart deploys to dev, staging, and prod with different values files. Rollback is one command. Version history is automatic. Never apply raw YAML manually in production.
The same Dockerfiles, upgraded
The Dockerfiles from Chapter 7 run unchanged on EKS. Docker Compose was the local dev version; Helm is the production version. Same images, different orchestrator.
Kubernetes restarts failed pods
If a pod crashes, K8s restarts it automatically. If a node fails, K8s reschedules the pods on a healthy node. This is the self-healing behaviour that makes Kubernetes worth the complexity.
Cost vs reliability trade-off
$80/month for always-on reliability vs $0 for your laptop. For a team of 5+ developers, $80/month is trivially worth it. For solo development, keep using Docker Compose locally.
