Four-Container Architecture

We run four containers — one for each service plus a local DynamoDB. The local DynamoDB replaces the real AWS service during development, keeping costs at $0.

The Most Important Optimisation: Layer Caching

Docker builds in layers. Copy dependency files before source code so packages are cached and only your code rebuilds on changes.

Your Daily Make Commands

make up
Build images and start all containers in correct order. Use every morning.
make down
Stop all containers. Data preserved in volumes.
make health
Curl all 3 health endpoints. Shows tick or cross for each.
make logs
Follow logs from all containers in one stream.
make logs-backend
Only agent backend logs. Use when debugging AI issues.
make rebuild
Force full no-cache rebuild. Use after adding packages.
make shell-mcp
Bash shell inside running MCP server. Debug tools.
make clean
Stop everything, delete volumes and images. Full reset.

What You Have Learned

Layer caching = free speed
Copy dependency files before source code. Once packages are cached, rebuilds take seconds. Order matters in every Dockerfile you write.
depends_on + healthcheck
depends_on alone only waits for start, not readiness. Add condition: service_healthy to wait for health checks before starting dependents.
Service names as DNS
Inside Docker Compose, containers talk by service name. http://neuron-mcp-server:8001 works from any container in the same network.
Never run as root
Always create a non-root user (UID 1000). If a container is compromised, a non-root process has limited blast radius.
Local DynamoDB = free dev
The local DynamoDB image is a full API replacement. Identical behaviour, zero cost, works offline. Always use it locally.
Same containers → EKS
The Dockerfiles you wrote today deploy to Kubernetes in Chapter 14. You are building the production foundation right now.
